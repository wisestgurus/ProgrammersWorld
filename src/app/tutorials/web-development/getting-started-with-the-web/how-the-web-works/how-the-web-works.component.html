Introduction

An information system made up of different websites linked together is referred to as the web. (An information system in a layman term is anything designed to collect, process, store, and distribute information).

Contents on the web are shared through the internet with the help of TCP/IP. However, the web also has its own protocol known as HTTP (which stands for HyperText Transfer Protocol) to handle specific web request made by the user to the web server. (i.e. a request-response model )

This request-response model is known as the communication of the web between the client and the server - client-server communiation. (i.e. The client makes a request to the server using a web browser and the server responds back. HTTP is the protocol that allows for such request-response communication). (A protocol is just a rule or a guide used in achieving anything)

HTTP establishes connection between the client and the server on the web to check whether the data requested exists. While TCP/IP ensures that the data returned by the web server is delivered successfully.

How HTTP works

For HTTP to establish connection between the client and the server, they both need to reference or point to the same location using a URL.

For instance, let's take this simple URL - (/api/books/) in bracket as an example.

  1. From the servers point of view
    • The server is constantly listening for any web client (or browser) that points to the URL above. If it gets a client request, then it responds back to the client with the corresponding data.
    • A web server is similar to a shop owner. If a customer requests for a particular item in the shop, then the shop owner gives the customer back the item as a response.
  2. From the clients (or web browsers) point of view
    • When the client points to the URL above, HTTP takes over. HTTP checks whether the URL specified by the client matches the one on the web server.
    • If it does, HTTP returns back a successful message to the browser, then TCP/IP trasmits the data returned by the server to the browser and finally the connection terminates (ends or closes).
    • If the URL doesn't match, HTTP instead returns back an error message to the browser, then the connection terminates with no data returned.
    • But wait, this means that the browser is stuck with diplaying nothing right? Luckily enough, web browsers have tackled such problem by instead displaying a "404 page not found" message.
Often times, we come across the "404 page not found" message, this is usually because the URL on the client side doesn't match the one on the server, which in turn doesn't exist and no data is returned.
HTTP messages are not visible on the homepage of a web browser. To be able to see it, you must open the dev tools.
Don't worry too much about the dev tools and HTTP messages - You will learn much more about them in their respective tutorial sections.
